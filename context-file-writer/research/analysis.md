# 综合分析

## 用户需求拆解

### 场景 1：项目新创建时

**用户痛点**：不知道从何入手，容易写得过多或过少

**解决方案**：
- 提供最小化模板，只包含必需内容
- 引导用户填写关键信息（技术栈、命令、约定）
- 解释每个部分的价值和写法

**输出**：初始 AGENTS.md（50-100 行）

### 场景 2：项目推进过程中

**用户痛点**：不知道什么时候更新，怎么更新

**解决方案**：
- 定义更新触发条件（AI 反复犯错、架构决策确定、新工具引入）
- 提供增量更新策略（不要整体重写）
- 提供 Agent 辅助更新的 prompt

**输出**：更新后的 AGENTS.md

### 场景 3：项目完成或阶段性完成

**用户痛点**：文件可能过时、冗余、矛盾

**解决方案**：
- 提供审查清单（准确性、相关性、完整性）
- 提供清理策略（删除过时、合并重复、分层组织）
- 定期审查节奏建议

**输出**：优化后的 AGENTS.md

## 补充的场景

### 场景 4：多工具统一

**问题**：团队使用 Claude Code、Cursor、Copilot 等多种工具

**解决方案**：
- 以 AGENTS.md 为单一事实来源
- 通过符号链接或导入让其他工具读取同一份内容
- 仅在需要工具特有功能时才添加工具专属文件

### 场景 5：不同项目类型适配

**问题**：前端、后端、Monorepo、CLI 等项目类型差异大

**解决方案**：
- 提供项目类型特定的模板
- 每个模板只包含该类型的关键约定
- 支持用户自定义扩展

### 场景 6：团队协作

**问题**：多人维护时容易产生矛盾和冗余

**解决方案**：
- 将 AGENTS.md 提交到 git
- 指定 owner 负责审查
- PR 模板添加检查项

## 核心设计原则

1. **引导式交互**：不是命令式生成，而是引导用户理解原则
2. **渐进式披露**：主文件简洁，详细内容按需加载
3. **质量保证**：基于研究数据的验证清单
4. **可扩展性**：支持项目类型、工具、规则的扩展
5. **全生命周期**：覆盖创建 → 更新 → 优化的完整流程

## 差异化设计

1. **基于研究数据**：引用 ETH Zurich、GitHub 2500+ 仓库等权威研究
2. **不只提供模板**：原则 + 模板 + 流程的完整体系
3. **多工具统一**：AGENTS.md 为单一事实来源
4. **内置质量检查**：基于数据的验证标准
