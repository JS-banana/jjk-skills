# 0001 — Demand Radar 采用精简 schema（13 字段 / 2 维优先级）

- 状态：Accepted
- 日期：2026-06-29
- 相关：`../design.md`（已废弃的富 schema 方案）、`../requirements.md` §1/§6、
  `plugins/jkk/skills/demand-radar/references/feishu-schema.md`、
  `plugins/jkk/skills/demand-radar/scripts/run.py`

## 背景

初版设计 `design.md`（2026-06-24）提出一套「富 schema」：约 19 个字段，外加 5 维
加权打分（痛点强度 / 出现频率 / 付费意愿 / 竞争缺口 / 实现难度，连续 1–5 分，加权求和
得「综合评分」），并配多个视图与评估元字段（评估备注 / 相关需求 / 标签 / 市场 等）。

落地后暴露问题（见 `requirements.md`）：

- 字段太多、大量字段长期为空、没有规范（§1）。
- 多个打分维度互相重复、定位不清（「需求强度、需求普遍性、可解决性、匹配度是否重复」，§6）。
- 数据质量一般，噪音多，偏离「面向万千大众的真实需求」的目标（§1、§4、§8）。

## 决策

采用**精简 schema**，并以 skill 为唯一权威：

- **13 个字段**：标题、一句话痛点、社区反响、需求方向、产品形态、需求强度、需求普遍性、
  优先级、处理状态、原文内容、原始链接、来源平台、语言。
- **2 维优先级**：`优先级 = f(需求强度 ∈ {强,中,弱} × 需求普遍性 ∈ {高,中,低})`，
  由 `scripts/run.py` 确定性派生；满 9 分且有 `P0证据` 才升 P0。
- **权威源**：schema 与校验逻辑只存在于 skill（`feishu-schema.md` + `run.py`）。
  `design.md` 仅作历史设计记录，不再是 schema 依据。

## 后果

- ✅ 字段精简、无空字段堆积、打分维度不再重复，契合「质量优先、宁缺毋滥」的目标。
- ✅ 校验集中在 `run.py`（确定性兜底），写入飞书前即可拦截脏数据。
- ⚠️ 放弃了 5 维连续打分的细粒度——这是**有意的取舍**：换取更高的数据质量与可维护性。
- ⚠️ `design.md` 与本 schema 不一致是**预期**的，不是缺陷。任何审计/重构若发现两者
  「冲突」，应以 skill 为准，而非回退到 `design.md`。
- 未来如需调整 schema，改动入口是 skill 的 `feishu-schema.md` + `run.py`，并在此追加新的 ADR。
