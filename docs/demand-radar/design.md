# 需求挖掘系统（Demand Radar）深度设计方案

> 基于3篇三木文章分析 + 用户现有基础设施 + 行业补充研究
> 设计日期：2026年6月24日

---

## 一、设计原则

1. **信号优先于噪音**：宁可漏掉一些需求，也不要推送大量无价值信息
2. **结构化优于自由文本**：每条需求必须有可量化、可排序的维度
3. **人机协作**：机器负责采集和初筛，人负责判断和决策
4. **复用现有基础设施**：Agent Reach + Hermes + OpenClaw + 飞书
5. **渐进式扩展**：先从高价值源开始，验证后再扩展

## 四、飞书多维表格Schema设计

### 4.1 表名：Demand Radar（需求雷达）

### 4.2 字段定义

| 字段名 | 类型 | 说明 | 必填 |
|--------|------|------|------|
| **标题** | Text | 需求的简短描述（≤50字） | ✅ |
| **需求描述** | Text | 详细的痛点描述 | ✅ |
| **用户原话** | Text | 原始用户表达（保留原始语言） | ✅ |
| **原始链接** | URL | 来源帖子/评论的链接 | ✅ |
| **来源平台** | SingleSelect | Reddit/PH/HN/V2EX/小红书/知乎/Twitter/GitHub/Exa/其他 | ✅ |
| **市场** | SingleSelect | 英文/中文/全球 | ✅ |
| **需求类别** | MultiSelect | 工具/效率/开发/设计/AI/营销/财务/健康/教育/生活/其他 | ✅ |
| **发现日期** | CreatedAt | 自动记录 | ✅ |
| **最后更新** | UpdatedAt | 自动记录 | ✅ |

### 4.3 评估维度字段（AI评分，1-5分）

| 字段名 | 类型 | 评分标准 |
|--------|------|---------|
| **痛点强度** | Number(1-5) | 1=轻微不便 3=明显痛点 5=严重阻碍 |
| **出现频率** | Number(1-5) | 1=单次提及 3=偶见 5=高频出现 |
| **付费意愿** | Number(1-5) | 1=免费替代 3=可接受付费 5=明确愿意付费 |
| **竞争缺口** | Number(1-5) | 1=成熟方案多 3=有方案但不满意 5=无专门方案 |
| **实现难度** | Number(1-5) | 1=复杂系统 3=中等 5=简单前端工具 |
| **综合评分** | Formula | =痛点强度×0.25 + 出现频率×0.20 + 付费意愿×0.25 + 竞争缺口×0.20 + 实现难度×0.10 |

### 4.4 状态管理字段

| 字段名 | 类型 | 选项 |
|--------|------|------|
| **处理状态** | SingleSelect | 待评估/评估中/有价值/待开发/开发中/已上线/已放弃 |
| **评估备注** | Text | 人工判断的补充说明 |
| **相关需求** | Text | 关联的其他需求记录 |
| **标签** | MultiSelect | 按主题自由打标 |

### 4.5 视图设计

| 视图名 | 类型 | 筛选/排序 | 用途 |
|--------|------|----------|------|
| **每日新增** | Table | 发现日期=今天 | 每天早上查看新发现 |
| **高价值待评估** | Table | 综合评分≥3.5 AND 处理状态=待评估 | 优先评估高分需求 |
| **英文市场** | Table | 市场=英文 | 按市场分别查看 |
| **中文市场** | Table | 市场=中文 | 按市场分别查看 |
| **看板视图** | Kanban | 按处理状态分组 | 跟踪需求处理进度 |
| **评分排行** | Table | 按综合评分降序 | 发现最高价值需求 |
| **热力分析** | Gallery | 按来源平台分组 | 分析哪个平台产出最多 |

### 5.2 每日采集脚本逻辑

**Step 1：数据采集（并行）**

```bash
# 英文市场
opencli reddit search "I wish there was" --subreddit SideProject,Entrepreneur,webdev,nocode,SaaS --limit 10 -f yaml
opencli reddit search "looking for a tool" --subreddit SideProject,Entrepreneur --limit 10 -f yaml

# Product Hunt（抓取每日页面）
curl -s "https://www.producthunt.com/" | jina

# Hacker News（API）
curl -s "https://hacker-news.firebaseio.com/v0/topstories.json" | head -30

# Twitter
twitter search "#buildinpublic looking for" --limit 10

# 中文市场
curl -s "https://www.v2ex.com/api/v2/topics/hot.json"
opencli xiaohongshu search "求推荐 工具" --limit 10 -f yaml
opencli xiaohongshu search "有没有好用的" --limit 10 -f yaml

# 语义搜索
exa search "user pain point for online tool" --num-results 10
```

**Step 2：AI分析（LLM处理）**

对每条采集到的原始数据，AI进行：
1. **痛点提取**：从用户原文中提取核心痛点
2. **去噪过滤**：排除已知成熟方案、广告、无关内容
3. **五维评分**：痛点强度/出现频率/付费意愿/竞争缺口/实现难度
4. **分类打标**：需求类别+市场+来源平台
5. **结构化输出**：生成符合飞书schema的记录

**Step 3：写入飞书**

```bash
lark-cli base +record-create \
  --base-token <NEW_BASE_TOKEN> \
  --table-id <TABLE_ID> \
  --fields '{"标题":"...", "需求描述":"...", ...}' \
  --as bot
```

### 5.3 过滤规则

**自动排除条件**（不进入飞书）：
- 用户原话为空或过于模糊（如"我想要好用的工具"）
- 已有明确成熟解决方案且无差异化空间
- 广告/推广/自我宣传内容
- 需求描述与AI/工具/产品完全无关
- 原始链接无法访问

**自动标记条件**：
- 同一需求在不同平台多次出现 → 标记"高频需求"
- 用户明确提到价格/付费 → 标记"付费意愿强"
- 竞品名称出现但用户表达不满 → 标记"竞品缺口"

### 5.4 去重策略

1. **URL去重**：同一链接不重复入库
2. **语义去重**：用AI判断新需求是否与已有需求本质相同
3. **跨平台关联**：同一需求在不同平台出现时，合并为一条记录，增加"出现频率"评分

## 七、关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 平台API限制 | 无法采集数据 | 随机间隔+备用源+Jina Reader兜底 |
| 信号质量低 | 推送大量无价值信息 | 严格过滤规则+人工反馈循环 |
| 需求重复 | 浪费评估时间 | URL+语义去重 |
| 平台反爬 | 账号风险 | 使用API而非爬虫+低频率 |
| AI评分不准 | 高价值需求被低估 | 人工校准+持续优化 |

## 八、与现有系统的关系

```
学习素材库（现有）          Demand Radar（新建）
├── 文章收藏               ├── 需求信号
├── AI活动                 ├── 痛点评分
├── 黑客松                 ├── 产品机会
└── 内容素材               └── 开发决策
     │                          │
     └──────────┬───────────────┘
                │
         互相参考关联
    （同一需求可关联到相关文章/活动）
```

两个系统独立运行但可互相参考：
- 学习素材库中发现的文章可能启发需求
- Demand Radar中的需求可关联到相关学习素材
- 黑客松活动可能直接对应某些需求方向
